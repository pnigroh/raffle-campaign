🎰 Raffle Campaign Manager

Django 4.2 · Unfold Admin

Overview

Raffle Campaign Manager is a Django application for running promotional campaigns and prize raffles. Staff create campaigns, attach prizes and submission codes, then share a public URL with participants. Once entries are collected, staff run a segmented raffle from the dashboard and export the winners.

✅
Default credentials (development): username admin · password admin123
⚠️
Change the default password before exposing the app to any network.

Features

📣

Campaign Management

Create campaigns with date ranges, active toggle, and configurable code validation.

🎫

Submission Codes

Bulk-import codes via CSV. Optionally require a valid code before saving a submission.

🌐

Public Form

Clean participant form — name, state, county, phone, email, and optional code field.

📊

Dashboard

Live overview of active campaigns, submission counts, and recent activity.

🎰

Segmented Raffles

Filter the pool by state, county, and date range. Live AJAX participant counter.

🏆

Multi-Prize Draws

Set winner quantities per prize in one draw. No duplicate winners across prizes.

📤

CSV Export

Export all submissions or raffle winners at any time.

🛠️

Unfold Admin

Modern, beautiful admin UI with a custom navigation sidebar and purple theme.

Installation

Requirements

  • Python 3.10+
  • pip
  • virtualenv (recommended)

Step-by-step

1 · Create & activate a virtual environment

python3 -m venv venv
source venv/bin/activate       # macOS / Linux
venv\Scripts\activate          # Windows

2 · Install dependencies

pip install -r requirements.txt

3 · Apply database migrations

python3 manage.py migrate

4 · Create a superuser

# Quick dev superuser (admin / admin123)
python3 manage.py create_superuser_default

# Or interactively
python3 manage.py createsuperuser

5 · Start the development server

python3 manage.py runserver

Then visit http://127.0.0.1:8000/dashboard/ and log in.

Schema Overview

Campaign ├── Prize (FK → Campaign) ├── SubmissionCode (FK → Campaign) ├── Submission (FK → Campaign) │ └── submission_code (OneToOne → SubmissionCode, optional) └── Raffle (FK → Campaign) └── RaffleWinner (FK → Raffle) ├── → Submission └── → Prize

A Campaign is the root entity. Everything else belongs to it. A Submission is one participant entry. A Raffle is a draw event that produces RaffleWinner records linking submissions to prizes.

start_dateDateTimeFieldWhen submissions open
end_dateDateTimeFieldWhen submissions close
is_activeBooleanFieldMaster on/off switch for the campaign
validate_submission_codeBooleanFieldIf True, a valid unused code is required before saving
allow_multiple_submissionsBooleanFieldIf False, the same email can only submit once

updated_atDateTimeField (auto)Last modification timestamp
💡
validate_submission_code controls the core mechanic: when True, the participant must enter a code that exists in SubmissionCode and has is_used=False. Once used, the code is marked as used and cannot be reused.

Public Submission Form

Share http://yourdomain.com/submit/<campaign-slug>/ with participants. The form collects:

  • First Name
  • Last Name
  • Submission Code — required if validate_submission_code is enabled, otherwise optional
  • State — US state dropdown
  • County
  • Phone
  • Email

Validation Rules

RuleCondition
Campaign must be activeis_active=True and current date is within start_date…end_date
Code must be validOnly when validate_submission_code=True
Code must be unusedSubmissionCode.is_used=False
No duplicate emailOnly when allow_multiple_submissions=False
✅
On success, the submission code is immediately marked as is_used=True and the participant sees a thank-you page.

Dashboard

The dashboard (/dashboard/) gives a bird's-eye view of all campaigns:

  • Stat cards: total campaigns, active campaigns, total submissions across all campaigns
  • Active campaigns table with submission counts, code stats, and quick-action links
  • Inactive campaigns table
  • Recent submissions feed (last 10 across all campaigns)

Campaign Detail

Clicking into a campaign shows:

  • Stat cards: total submissions, available codes, used codes, number of prizes
  • Filter bar — filter by state, county, or free-text search
  • Paginated submissions table
  • Prizes list
  • Raffle history with winner counts and export links
  • Buttons: Export Submissions, Import Codes, Run Raffle

Running a Raffle

  1. Ensure the campaign has at least one Prize configured (via Admin or admin panel).
  2. Go to Dashboard → Campaign → Run Raffle.
  3. Apply segment filters to narrow the participant pool:
    • State — only participants from a specific US state
    • County — partial match on county name
    • Submitted From / To — date range filter
  4. The live participant counter updates via AJAX as you change filters.
  5. Set the winner quantity for each prize (leave at 0 to skip a prize).
  6. Add optional notes for auditing purposes.
  7. Click Confirm & Run Raffle.
  8. View the results page — winners grouped by prize with position badges.
  9. Click Export Winners CSV to download the results.
💡
How the draw works: the filtered submission pool is shuffled with Python's random.shuffle() (a Fisher-Yates shuffle using the Mersenne Twister PRNG). Winners are drawn in order across prizes — once a submission wins one prize, it cannot win another in the same raffle.
⛔
Raffles are permanent records. Each draw creates a new Raffle object. Running the raffle multiple times will create multiple independent draws — there is no undo.

Exporting Data

Submissions Export

From the Campaign Detail page, click Export CSV. If state/county filters are active, only the filtered rows are exported.

Columns: First Name, Last Name, Email, Phone, State, County, Submission Code, Submitted At

Winners Export

From any Raffle Results page, click Export Winners CSV.

Columns:

Prize, Position, First Name, Last Name, Email, Phone, State, County, Submission Code, Submitted At

Production Checklist

  • Set DEBUG=False in .env
  • Generate and set a strong SECRET_KEY
  • Set ALLOWED_HOSTS to your domain(s)
  • Switch DATABASES to PostgreSQL
  • Run python3 manage.py collectstatic
  • Serve static files via nginx or whitenoise
  • Configure HTTPS / SSL certificate
  • Change the default admin password
  • Set up regular database backups
  • Review AUTH_PASSWORD_VALIDATORS settings

Environment Variables

.envSECRET_KEY=your-very-long-random-secret-key
DEBUG=False
ALLOWED_HOSTS=yourdomain.com,www.yourdomain.com
⚠️
The default SQLite database is fine for development and low-traffic use, but switch to PostgreSQL for production reliability and concurrency.
